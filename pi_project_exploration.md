# 圆周率纪念海报网站 (Pi Birthday Poster) - 技术调研与架构方案

## 1. 项目背景与竞品分析
**项目目标**：打造一个让用户可以在圆周率（$\pi$）中寻找具有特殊意义的数字（如 8 位数生日 `YYYYMMDD`、纪念日、手机尾号等），并生成极具美感、带上下文的专属纪念海报，用于社交分享。

**市面竞品现状**：
- **功能缺失**：现有的 Pi 查询网站（如 MyPiDay, The Pi-Search Page, Calculat.io）大多停留在“极客查询工具”阶段，界面简陋，完全缺乏**生成纪念图片**和**社交分享**功能。
- **数据与格式限制**：部分网站只能查询 `MMDDYY`（6位）格式的生日，或者查询深度极浅（仅前100万位）。
- **市场机会**：将冷冰冰的数学常量包装为“宇宙级浪漫”（例如：“在无限不循环的宇宙里，我们相遇在第3,141,592位”），具有极强的病毒传播和社交裂变潜力。

---

## 2. 数据量级与命中率测算
在测试阶段，我们发现寻找长度不同的数字串，对 Pi 的数据深度要求完全不同：
- **6位数字（如 `980512`）**：在 **1,000万位（10 Million）** 数据中极大概率能找到。
- **8位全格式生日（如 `19980512`）**：在 1,000 万位中命中率不足 10%。为了保证 >99% 的命中率，**必须使用前 10 亿位（1 Billion，约 1GB 大小）** 的圆周率数据。

---

## 3. 核心查询技术测算与对比
针对 10 亿位（1GB）数据的查询，我们进行了深度测试。目标是保证低端设备和弱网环境下延迟小于 6 秒。

### 方案 A：第三方 API 调用（放弃）
- 测试了 `angio.net` 和 `pi.delivery` 等 API。
- **结论**：免费 API 极不稳定（出现 500 错误），或者限制每次只能获取 1000 位，无法满足全量高速检索需求，延迟和稳定性均不可控。

### 方案 B：本地文件纯内存搜索（性能极佳，但吃内存）
我们生成了 1GB 的模拟文件，分别用 Node.js 和 Python 进行了测试：
1. **Node.js (Buffer 检索)**
   - 文件加载：~600ms
   - 查询耗时：~630ms
   - 总耗时：~1.2秒（需消耗 >1GB 内存）
2. **Python (mmap 内存映射)**
   - 文件加载：~0ms（零拷贝技术）
   - 查询耗时：~820ms
   - 总耗时：< 1秒（内存占用极低）

**结论**：即使是 10 亿位的数据，单次检索耗时也能稳定控制在 1.5 秒内。

## 4. 生产级部署架构设计（国内访问极速方案）

考虑到开发者希望使用 **GitHub Actions/Vercel 部署**，并且域名托管在 **Cloudflare**，同时目标受众主要在**中国大陆**（需支持低端机、弱网，延迟 < 6s）。我们设计了以下极其精巧的“零成本/高可用”架构：

### 4.1 架构痛点
- **Vercel Serverless 限制**：Vercel 的无服务器函数最大内存通常为 1024MB，且代码包大小限制为 250MB，**绝无可能直接打包和加载 1GB 的圆周率文件**。
- **国内网络环境**：直接访问 Vercel 域名（`.vercel.app`）或 IP 往往被墙，下载大文件极易超时。

### 4.2 最终架构选型：Vercel + Cloudflare + OSS 分块检索

#### 📍 前端：本地轻量级渲染 (React + Vite)
- **部署**：托管于 Vercel。
- **网络优化**：通过 Cloudflare 绑定自定义域名，并开启 Proxied（小黄云），完美绕过国内对 Vercel 的 SNI 阻断。
- **海报生成**：**绝对不要在后端生成图片！** 图片体积大，弱网下传输极慢。采用前端 `html2canvas` 或 `dom-to-image` 在用户手机本地利用 Canvas 动态渲染海报，做到“秒生成”，节省服务器带宽。

#### 📍 存储：对象存储 (Cloudflare R2 / 阿里云 OSS / 腾讯云 COS)
- 将 1GB 的圆周率文本文件（如 `pi_1B.txt`）上传至廉价的对象存储中。国内推荐阿里云/腾讯云（内网互通或 CDN 加速），海外可用完全免费的 Cloudflare R2。

#### 📍 后端：Vercel Serverless API + HTTP Range 请求 (核心黑科技)
为了突破 Vercel 内存和打包限制，采用 **“预构建索引 + 分块下载”** 策略：
1. **构建索引**：在本地预处理 1GB 文件，每隔 1,000 万位抽取一段数字作为“路标”，生成一个极小的 `index.json`（仅几百 KB），打包进 Vercel。
2. **极速检索**：
   - 用户查询 `19980512`，Vercel API 瞬间读取 `index.json`，预判该数字大致落在 1GB 文件的哪个区间（例如 `30MB - 40MB` 处）。
   - Vercel API 向对象存储发送带有 `Range: bytes=30000000-40000000` 头的 HTTP 请求，**仅拉取这 10MB 的数据块**。
   - 在这 10MB 的内存中执行字符串查找，截取前后 30 位上下文，组装成极简 JSON（~100 字节）返回给前端。
3. **性能预期**：整个网络请求 + 云端检索链路的数据传输量极小，耗时稳定在 2~3 秒内，完美解决弱网和低端设备的性能瓶颈。
